ch6. VACUUM Processing

ch6. VACUUM Processing

VACUUM处理是一个维护进程,可以确保PostgreSQL持续稳定运行。它主要有两个任务:
删除死元组和冻结事务ID,这两点在第5.10节中简要提到过。

为了删除死元组,VACUUM处理提供了两种模式:并发VACUUM和完全VACUUM。
并发VACUUM(通常直接称为VACUUM)会逐页删除表文件中的死元组,其他事务在此过程中仍然可以读取该表。
而完全VACUUM则会删除整个文件中的死元组并对活元组进行碎片整理,在此过程中其他事务无法访问该表。

尽管VACUUM处理对PostgreSQL至关重要,但相比其他功能其改进进度一直较慢。
例如在8.0版本之前,这一进程必须手动执行(使用psql工具或cron守护进程)。直到2005年才实现了自动化,
引入了autovacuum守护进程。

由于VACUUM处理需要扫描整个表,因此是一个耗费资源的过程。
在8.4版本(2009年)中引入了可见性映射(Visibility Map,VM)来提高删除死元组的效率。
在9.6版本(2016年)中,冻结进程通过增强VM得到了进一步改进。

6.1 Outline of Concurrent VACUUM

VACUUM处理会对指定的表或整个数据库中的所有表执行以下任务:

1、删除死元组
删除每个页面中的死元组并对活元组进行碎片整理。
删除指向死元组的索引元组。
冻结旧的事务ID

2、必要时冻结元组的旧事务ID。
更新与冻结事务ID相关的系统目录(pg_database和pg_class)。
如果可能,删除日志文件(clog)的不必要部分。
其他

3、更新已处理表的空间管理(FSM)和可见性映射(VM)。
更新若干统计信息(pg_stat_all_tables等)。
假定读者熟悉以下术语:死元组、冻结事务ID、FSM和日志文件(clog);如果不熟悉,请参考第5章。
VM在第6.2节中介绍。

以下是VACUUM处理的伪代码:

Code Pseudocode: Concurrent VACUUM

(1) 从指定的表中获取每个表。
(2) 为目标表获取一个ShareUpdateExclusiveLock锁。这个锁允许其他事务进行读取操作。
(3) 扫描所有页面以获取所有死元组,并在必要时冻结旧的事务 ID。
(4) 删除指向相应死元组的索引元组(如果存在)。
(5) 对表的每个页面执行以下任务,第(6)和(7)步。
(6) 删除死元组,并重新分配页面中的活元组。
(7) 更新目标表的相应 FSM 和 VM。
(8) 通过 index_vacuum_cleanup()@indexam.c 函数清理索引。
(9) 如果最后一个页面没有任何元组,则截断该页面。
(10) 更新与目标表的vacuum处理相关的统计信息和系统目录。
(11) 更新与vacuum处理相关的统计信息和系统目录。
(12) 如果可能,删除日志文件(clog)中不必要的文件和页面。

这个伪代码有两个部分：一个是针对每个表的循环,另一个是后处理。内部循环可以分为三个块。
每个块都有各自的任务。

这三个块和后处理过程概括如下:

PARALLEL option

自 PostgreSQL 13 版本开始,VACUUM 命令支持 PARALLEL 选项。如果设置了此选项,
并且创建了多个索引,则 vacuuming 索引和清理索引的阶段会并行处理。

需要注意的是,这个功能只适用于 VACUUM 命令,autovacuum 不支持此功能。

6.1.1. First Block

这个块执行冻结处理,并删除指向死元组的索引元组。

首先,PostgreSQL 扫描目标表以构建死元组的列表,并尽可能冻结老元组。
该列表存储在名为 maintenance_work_mem 的本地内存中。冻结处理在第 6.3 节中有描述。

扫描完成后,PostgreSQL 通过参考死元组列表来删除索引元组。这个过程在内部称为”清理阶段”。
这是一个代价高昂的过程,所以 PostgreSQL 在 11 版本中进行了改进。

在 10 版本或更早的版本中,清理阶段总是执行。在 11 版本或更新的版本中,如果目标索引是 B 树,
是否执行清理阶段由配置参数 vacuum_cleanup_index_scale_factor 决定。

请参阅该参数的详细描述。

如果 maintenance_work_mem 已满且扫描未完成,PostgreSQL 将进入下一个任务,
即步骤(4)到(7)。然后,它会回到步骤(3),继续剩余的扫描。

6.1.3. Third Block

第三个块执行索引删除后的清理操作,并更新与 vacuum 处理相关的统计信息和系统目录,针对每个目标表。

此外,如果最后一页没有元组,它将从表文件中截断。

6.1.4. Post-processing

当 vacuum 处理完成后,PostgreSQL 会更新所有与 vacuum 处理相关的统计信息和系统目录。
如果可能的话,它还会删除 clog 中不必要的部分(第 6.4 节)。

Ring Buffer

Vacuum 处理使用了一个环形缓冲区,这在第 8.5 节有描述。
因此,已经处理的页面不会被缓存在共享缓冲区中。

6.2 Visibility Map

Vacuum 处理是一个代价高昂的操作。因此,在 8.4 版本中引入了 VM(可见性映射)来降低这种成本。

VM 的基本概念很简单。每个表都有一个独立的可见性映射,用于保存表文件中每个页面的可见性状态。
页面的可见性决定了该页面是否包含死元组。Vacuum 处理可以通过使用相应的可见性映射(VM)
跳过没有死元组的页面。

图 6.2 展示了 VM 的使用方式。

的 VM 保存了哪些页面包含死元组的信息。在这种情况下, Vacuum 处理可以通过查看 VM 的信息,
跳过第 1 页。

每个 VM 由一个或多个 8 KB 的页面组成,并以 ‘vm’ 后缀存储在文件系统中。

6.3 Freeze Processing

冻结处理有两种模式。为了方便区分,它们被称为lazy mode和eager mode。
根据特定条件，它可以以任一模式执行。

Note

并发 VACUUM 在内部通常被称为”懒惰 vacuum”。
但是,本文中定义的懒惰模式是冻结处理的一种模式。

冻结处理通常以惰性模式运行，但当满足特定条件时就会运行急切模式。

在懒惰模式下,冻结处理只扫描包含死元组的页面,使用目标表的相应 VM。

相反,在积极模式下,它会扫描所有页面,而不管每个页面是否包含死元组。
它还会更新与冻结处理相关的系统目录,并在可能的情况下删除 clog 中不必要的部分。

第 6.3.1 节和第 6.3.2 节分别描述了这两种模式。
第 6.3.3 节描述了如何在积极模式下改进冻结过程。

FREEZE option

带有 FREEZE 选项的 VACUUM 命令会强制冻结指定表中的所有 txid。
这是在积极模式下执行的,但是 freezeLimit 被设置为 OldestXmin
(而不是 ‘OldestXmin - vacuum_freeze_min_age’)。例如,
当 VACUUM FULL 命令由 txid 5000 执行且没有其他正在运行的事务时,OldestXmin
被设置为 5000,小于 5000 的 txid 都会被冻结。

6.3.3. Improving Freeze Processing in Eager Mode

在9.5或更早版本中的eager模式并不高效,因为它总是扫描所有页面。
比如在6.3.2节的例子中,即使0号页面中的所有元组都已经被冻结,也还是会被扫描。

为了解决这个问题,在9.6版本中对VM和冻结过程进行了改进。
如6.2.1节所述,新的VM包含了每个页面中是否所有元组都已冻结的信息。
在eager模式下执行冻结处理时,仅包含已冻结元组的页面可以被跳过。

图6.6展示了一个例子。在冻结这个表时,通过参考VM的信息,0号页面被跳过了。在冻结完1号页面之后,
相关的VM信息也被更新,因为该页面的所有元组都已被冻结。

6.5 Autovacuum Daemon

在 PostgreSQL 中,Vacuum 处理已经通过自动 vacuum 后台进程自动化了,
使得 PostgreSQL 的操作变得非常简单。

自动 vacuum 后台进程会定期启动多个自动 vacuum 工作进程。默认情况下,
它每 1 分钟(由 autovacuum_naptime 定义)唤醒一次,并启动三个工作进程
(由 autovacuum_max_works 定义)。

由自动 vacuum 后台进程启动的自动 vacuum 工作进程会并发地为
各自的表执行 vacuum 处理,逐步并且对数据库活动的影响很小。

6.5.1. Conditions for autovacuum to run

自动 vacuum 进程会在以下任意条件满足时为目标表运行:

1、当前 txid 先于以下表达式:

relfrozenxid + autovacuum_freeze_max_age

其中 relfrozenxid 是定义在 pg_class 中的目标表的 relfrozenxid 值,
autovacuum_freeze_max_age 是一个配置参数(默认为 200,000,000)。
如果满足这个条件,则自动 vacuum 进程会对目标表执行冻结处理。

2、死元组数量大于以下表达式:

autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples

其中 autovacuum_vacuum_threshold(默认为 50)和 autovacuum_vacuum_scale_factor(默认为 0.2)是配置参数,
reltuples 是目标表的元组数。
例如,如果目标表有 10,000 个元组和 2,100 个死元组,则自动
vacuum 进程会运行,因为

2,100 > 50 + 0.2 × 10,000。

3、插入的元组数量大于以下表达式:

autovacuum_vacuum_insert_threshold + autovacuum_vacuum_insert_scale_factor × reltuples

其中 autovacuum_vacuum_insert_threshold(默认为 1000)和 autovacuum_vacuum_insert_scale_factor(默认为 0.2)
是配置参数, reltuples 是目标表的元组数。
例如,如果目标表有 10,000 个元组和 3,010 个插入元组,
则自动 vacuum 进程会运行,因为

3,010 > 1000 + 0.2 × 10,000。

这个条件是从 13 版本开始添加的。

此外,如果以下条件对目标表满足,自动 vacuum 进程还将执行analyze处理:

mod_since_analyze > autovacuum_analyze_threshold + autovacuum_analyze_scale_factor × reltuples

其中 mod_since_analyze 是上次分析处理后对表执行的修改次数(INSERT、DELETE 或 UPDATE)。
autovacuum_analyze_threshold(默认为 50)和 autovacuum_analyze_scale_factor(默认为 0.1)
是配置参数, reltuples 是目标表的元组数。

例如,如果目标表有 10,000 个元组,且自上次分析处理后修改了 1,100 个元组,
则自动 vacuum 进程会运行,因为

1,100 > 50 + 0.1 × 10,000。

Info

relation_needs_vacanalyze()函数负责确定目标表是否需要执行 vacuum 或分析操作。

6.6 Full VACUUM

尽管并发 VACUUM 对于操作至关重要,但它并不够充分。例如,即使删除了许多死元组,
它也无法减小表的大小。

图 6.8 展示了一个极端的例子。假设一个表由三个页面组成,每个页面包含六个元组。执
行以下 DELETE 命令删除元组,并执行 VACUUM 命令删除死元组:

虽然死元组被移除了,但表的大小并没有减小。这不仅浪费了磁盘空间,还对数据库性能产生负面影响。
比如在上述示例中,当读取表中的三个元组时,必须从磁盘加载三个页面。

为了处理这种情况,PostgreSQL 提供了完全 VACUUM 模式。图 6.9 概括了这种模式的工作原理。

[1] 创建新的表文件: 图 6.9(1)
当对一个表执行 VACUUM FULL 命令时,PostgreSQL 首先为该表获取 AccessExclusiveLock 锁,
并创建一个大小为 8 KB 的新表文件。AccessExclusiveLock 锁阻止其他用户访问该表。

[2] 将活元组复制到新表: 图 6.9(2)
PostgreSQL 只将旧表文件中的活元组复制到新表。

[3] 删除旧文件,重建索引,并更新统计信息、FSM 和 VM: 图 6.9(3)
在复制所有活元组后,PostgreSQL 删除旧文件,重建所有相关的表索引,
更新该表的 FSM 和 VM,并更新相关的统计信息和系统目录。

完全 VACUUM 的伪代码如下所示:

Pseudocode: Full VACUUM

(1) FOR each table
(2) Acquire AccessExclusiveLock lock for the table
(3) Create a new table file
(4) FOR each live tuple in the old table
(5) Copy the live tuple to the new table file
(6) Freeze the tuple IF necessary
END FOR
(7) Remove the old table file
(8) Rebuild all indexes
(9) Update FSM and VM
(10) Update statistics
Release AccessExclusiveLock lock
END FOR
(11) Remove unnecessary clog files and pages if possible

在使用 VACUUM FULL 命令时,需要考虑以下两点:

  1. 当完全 VACUUM 正在处理时,没有人可以访问(读取/写入)该表。
  2. 最多使用该表两倍的磁盘空间临时存储;因此,在处理大型表时,需要检查剩余的磁盘容量。
