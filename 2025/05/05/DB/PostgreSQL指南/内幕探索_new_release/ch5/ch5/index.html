ch5. 并发控制

CONCURRENCY CONTROL

并发控制是在数据库中多个事务并发运行时，维护ACID属性中原子性和隔离性的一种机制。

并发控制主要有三种广泛的技术：多版本并发控制（MVCC）、
严格两阶段锁定（S2PL）和乐观并发控制（OCC）。每种技术都有多种变体。

在MVCC中，每次写操作都会创建数据项的新版本，同时保留旧版本。当事务读取数据项时，系统会选择一个版本以确保
单个事务的隔离性。MVCC的主要优点在于“读者不会阻塞写者，写者也不会阻塞读者”，
相比之下，例如基于S2PL的系统在写入数据项时必须阻塞读者，因为写者会为此数据项获取排他锁。
PostgreSQL和一些关系型数据库管理系统使用了MVCC的一个变体，称为快照隔离（SI）。

为了实现SI，一些数据库系统如Oracle使用回滚段。在写入新数据项时，数据项的旧版本会被写入回滚段，
随后新项会覆盖到数据区域。PostgreSQL采用了更简单的方法。新数据项直接插入到相关的表页中。
在读取项时，PostgreSQL通过应用可见性检查规则，为每个事务选择数据项的适当版本。

SI不允许出现ANSI SQL-92标准中定义的三种异常：脏读、不可重复读和幻读。然而，SI不能达到真正的可串行化，
因为它允许如写偏斜和只读事务偏斜等串行化异常。需要注意的是，基于经典可串行化定义
的ANSI SQL-92标准与现代理论中的定义并不等价。

为了解决这个问题，从9.1版开始引入了可串行化快照隔离（SSI）。SSI可以检测串行化异常并解决由此引起的冲突 ，
因此PostgreSQL 9.1或更高版本提供了真正的SERIALIZABLE隔离级别。
（另外，SQL Server也使用SSI；Oracle仍然仅使用SI。）

5.1 事务ID

每当一个事务开始时，事务管理器会为其分配一个唯一的标识符，称为事务ID（txid）。
PostgreSQL中的txid是一个32位无符号整数，大约为42亿（即数十亿）。如果你在一个事务开始后执行内置
的txid_current()函数，该函数会返回当前的txid，如下所示：

PostgreSQL保留了以下三个特殊的txid：

0 表示无效的txid。

1 表示引导txid，仅在数据库集群初始化时使用。

2 表示冻结txid，这个概念在5.10.1节中有描述。

txid之间可以相互比较。例如，从txid 100的角度来看，大于100的txid处于“未来”，

由于在实际系统中txid空间可能不够用，PostgreSQL将txid空间视为一个循环。前21亿个txid被视为“过去”，接下来的21亿个txid被视为“未来”（图5.1b）。

note

请注意，BEGIN命令本身并不会被分配一个txid。

在PostgreSQL中，当执行完BEGIN命令后，首次执行命令时，事务管理器才会分配一个txid，随后事务才真正开始。

5.2 元组结构

表页面中的堆元组分为两种类型：常规数据元组和TOAST元组。本节仅描述常规元组。

一个堆元组由三部分组成：HeapTupleHeaderData结构、NULL位图以及用户数据（图5.2）。


The HeapTupleHeaderData structure is defined in src/include/access/htup_details.h.

HeapTupleHeaderData结构包含七个字段，但在后续章节中只需要其中的四个：

t_xmin 存储插入此元组的事务的txid。

t_xmax 存储删除或更新此元组的事务的txid。
如果这个元组未被删除或更新，t_xmax 则被设置为0，表示无效（INVALID）。

t_cid 持有命令ID（cid），它是当前事务中在此之前执行的SQL命令数量，从0开始计数。
例如，假设我们在一个事务中执行三条INSERT命令：BEGIN; INSERT; INSERT; INSERT; COMMIT;。
如果第一条命令插入了这个元组，t_cid就被设置为0。如果是第二条命令插入，则t_cid设为1，以此类推。

t_ctid 持有指向自身或新元组的元组标识符（tid）。
tid（在1.3节中描述）用于在表内唯一标识一个元组。
当这个元组被更新时，此元组的t_ctid会指向新元组；否则，t_ctid指向自己。

本节介绍元组如何被插入、删除和更新。接着，简要描述用于插入和更新元组的空闲空间映射
（Free Space Map, FSM）。

5.3.1. Insertion

在插入操作中，新元组会直接被插入到目标表的一个页面中（如图5.4所示）。

假设有一个事务，其txid为99，该事务在一个页面中插入了一个元组。
在这种情况下，插入元组的头字段被设置如下：

Tuple_1:

t_xmin 设置为99，因为这个元组是由txid 99插入的。
t_xmax 设置为0，因为这个元组尚未被删除或更新。

5.3.2. Deletion

在删除操作中,目标元组是逻辑删除的。执行 DELETE 命令的 txid 值被设置为元组的 t_xmax 字段(如图 5.5 所示)。

假设 Tuple_1 被 txid 111 删除。在这种情况下, Tuple_1 的头部字段会被设置如下:

Tuple_1:
t_xmax 被设置为 111。

如果 txid 111 已提交,那么 Tuple_1 就不再需要了。通常在 PostgreSQL 中,这种不需要的元组被称为”死元组”。

这些死元组最终应该从页面中删除。清理死元组的过程称为 VACUUM 处理,将在第 6 章中介绍。

5.3.3. Update

在更新操作中,PostgreSQL 会逻辑地删除最新的元组,并插入一个新的元组(如图 5.6 所示)。

假设一个记录被 txid 99 插入,然后被 txid 100 更新了两次。

当第一个 UPDATE 命令执行时,Tuple_1 会被逻辑删除,通过将 t_xmax 设置为 100。然后会插入一个新的 Tuple_2。Tuple_1 和 Tuple_2 的头部字段如下:

Tuple_1:
t_xmax 被设置为 100
t_ctid 被重写为 (0,2)

Tuple_2:
t_xmin 被设置为 100
t_xmax 被设置为 0
t_cid 被设置为 0
t_ctid 被设置为 (0,2)

当第二个 UPDATE 命令执行时,Tuple_2 会被逻辑删除,并插入一个新的 Tuple_3。Tuple_2 和 Tuple_3 的头部字段如下:

Tuple_2:
t_xmax 被设置为 100
t_ctid 被重写为 (0,3)

Tuple_3:
t_xmin 被设置为 100
t_xmax 被设置为 0
t_cid 被设置为 1
t_ctid 被设置为 (0,3)

如果 txid 100 被提交,Tuple_1 和 Tuple_2 将成为死元组。如果 txid 100 被中止,Tuple_2 和 Tuple_3 将成为死元组。

5.3.4. Free Space Map

当插入 heap 或索引元组时,PostgreSQL 会使用相应表或索引的 FSM (Free Space Map) 来选择可以插入的页面。

如前文 1.2.3 节所提到的,所有表和索引都有各自的 FSM。每个 FSM 存储了相应表或索引文件中每个页面的空闲空间容量信息。

所有的 FSM 都以 ‘fsm’ 为后缀存储,并且在需要时会被加载到共享内存中。

5.4 提交日志（clog）

PostgreSQL 将事务的状态存储在提交日志(Commit Log)中。提交日志,通常称为 clog,分配在共享内存中,并在整个事务处理过程中使用。

本节介绍了 PostgreSQL 中事务的状态、clog 的操作以及 clog 的维护。

5.4.1. Transaction Status

PostgreSQL 定义了四种事务状态: IN_PROGRESS、COMMITTED、ABORTED 和 SUB_COMMITTED。

前三种状态是不言自明的。例如,当一个事务正在进行时,其状态为 IN_PROGRESS。

SUB_COMMITTED 是针对子事务的状态,在本文中没有对其作进一步描述。

5.4.2. How Clog Performs

clog 由一个或多个 8 KB 的页面组成,存放在共享内存中。它在逻辑上形成一个数组,
数组的索引对应于相应的事务 ID,数组中的每一项都保存着对应事务 ID 的状态。图 5.7 展示了 clog 的结构及其工作原理。

当当前的事务 ID 增加到 clog 无法再存储时,就会追加一个新的页面。

当需要获取事务的状态时,会调用内部函数。这些函数会读取 clog,并返回所请求事务的状态。
(也可参考第 5.7.1 节中的”提示位”部分)

5.4.3. Maintenance of the Clog

当 PostgreSQL 关闭或者检查点进程运行时,clog 的数据会被写入存储在 pg_xact 子目录下的文件中。
(需要注意,在 9.6 或更早版本中,pg_xact 被称为 pg_clog。)这些文件被命名为 ‘0000’、’0001’ 等。

每个文件的最大尺寸为 256 KB。例如,如果 clog 使用了 8 个页面(从第一个页面到第八个页面,总大小为 64 KB),它的数据就会被写入
‘0000’(64 KB)。如果 clog 使用了 37 个页面(296 KB),它的数据就会被写入 ‘0000’ 和 ‘0001’，分别为 256 KB 和 40 KB。

当 PostgreSQL 启动时,存储在 pg_xact 文件中的数据会被加载以初始化 clog。

clog 的大小会持续增加,因为每当 clog 被填满时都会追加一个新的页面。但并非 clog 中的所有数据都是必需的。
在第 6 章中描述的 Vacuum 处理会定期删除这些旧数据(包括 clog 页面和文件)。

有关删除 clog 数据的详细信息,请参阅第 6.4 节。

5.5 事务快照

事务快照是一个数据集,它存储了某一特定时间点所有事务是否处于活动状态的信息。这里,所谓”活动事务”是指正在进行或尚未开始的事务。

PostgreSQL 内部将事务快照的文本表示格式定义为 ‘100:100:’。例如, ‘100:100:’ 表示”小于 99 的事务 ID 不是活动的,大于或等于 100 的事务 ID 是活动的”。

在下面的描述中,将使用这种方便的表示形式。如果你不熟悉这种表示法,请参见下面的说明。

PostgreSQL 提供了一个内置函数 pg_current_snapshot,它可以返回当前会话的事务快照,并以文本表示形式输出。

事务快照的文本表示形式是’xmin:xmax:xip_list’，各个组成部分说明如下:

xmin
(仍在活动的最早事务 ID)：所有早于这个 ID 的事务要么已提交并可见, 要么已回滚并死掉。

xmax
(尚未分配的第一个事务 ID)：所有大于或等于这个 ID 的事务在快照时刻还未开始, 因此对当前快照是不可见的。

xip_list
(快照时刻正在活动的事务 ID列表)：列表只包含 xmin 和 xmax 之间的活动事务 ID。

例如, 在快照 ‘100:104:100,102’ 中, xmin 是 ‘100’, xmax 是 ‘104’, xip_list 是 ‘100,102’。

Here are two specific examples:

第一个示例 ‘100:100:’ 意味着以下情况(图 5.8(a)):
小于或等于 99 的事务 ID 都不是活动的,因为 xmin 是 100。
大于或等于 100 的事务 ID 都是活动的,因为 xmax 是 100。

第二个示例 ‘100:104:100,102’ 意味着以下情况(图 5.8(b)):
小于或等于 99 的事务 ID 都不是活动的。
大于或等于 104 的事务 ID 都是活动的。
事务 ID 100 和 102 是活动的,因为它们在 xip 列表中,而事务 ID 101 和 103 是不活动的。

事务快照由事务管理器提供。在 READ COMMITTED 隔离级别下, 每执行一条 SQL 命令时都会获取一个快照; 否则(REPEATABLE READ 或 SERIALIZABLE),
事务只在第一条 SQL 命令执行时获取一个快照。获取的事务快照用于对元组进行可见性检查,在第 5.7 节中有描述。

在使用获取的快照进行可见性检查时, 快照中的活动事务必须被视为正在进行中,即使它们实际上已经提交或回滚。这一规则很重要,
因为它造成了 READ COMMITTED 和 REPEATABLE READ (或 SERIALIZABLE) 之间行为的差异。我们将在后续章节中多次提到这一规则。

在本节的其余部分, 我们将使用特定的场景图 5.9 来描述事务管理器和事务。

事务管理器总是持有当前正在运行的事务的信息。假设有三个事务依次启动,其中 Transaction_A 和 Transaction_B 的隔离级别是
READ COMMITTED, Transaction_C 的隔离级别是 REPEATABLE READ。

T1:
Transaction_A 启动并执行第一条 SELECT 命令。在执行第一条命令时, Transaction_A 请求了当前时刻的事务 ID 和快照。
在这个场景中, 事务管理器分配了事务 ID 200, 并返回了事务快照 ‘200:200:’。

T2:
Transaction_B 启动并执行第一条 SELECT 命令。事务管理器分配了事务 ID 201, 并返回了事务快照 ‘200:200:’ ,
因为 Transaction_A (事务 ID 200) 正在进行中。因此, Transaction_A 对 Transaction_B 是不可见的。

T3:
Transaction_C 启动并执行第一条 SELECT 命令。事务管理器分配了事务 ID 202, 并返回了事务快照 ‘200:200:’,
因此 Transaction_A 和 Transaction_B 对 Transaction_C 是不可见的。

T4:
Transaction_A 已经提交。事务管理器删除了关于这个事务的信息。

T5:
Transaction_B 和 Transaction_C 执行各自的 SELECT 命令。
Transaction_B 需要一个事务快照,因为它处于 READ COMMITTED 级别。在这个场景中, Transaction_B 获得了一个新的快照 ‘201:201:’,
因为 Transaction_A (事务 ID 200) 已经提交。因此, Transaction_A 对 Transaction_B 不再是不可见的。Transaction_C 不需要事务快照,
因为它处于 REPEATABLE READ 级别, 使用之前获得的快照 ‘200:200:’。因此, Transaction_A 对 Transaction_C 仍然是不可见的。

5.6 可见性检查规则

可见性检查规则是一系列规则，用于根据元组的t_xmin和t_xmax、clog（提交日志）以及获取的事务快照来判断每
个元组是否可见。

这些规则过于复杂，无法详细解释。因此，本文档仅展示后续描述所需的最简规则。在以下内容中，
我们将忽略与子事务相关的规则，并且不讨论t_ctid，即不考虑在单个事务中被更新超过两次的元组。

选定的规则共有十条，它们可以分为三种情况。

5.6.1. Status of t_xmin is ABORTED

当元组的t_xmin状态为ABORTED时，该元组总是不可见的（规则1），因为插入这个元组的事务已经被中止了。

这条规则具体地表达为以下数学表达式。

Rule 1: If Status(t_xmin) = ABORTED ⇒ Invisible

5.6.2. Status of t_xmin is IN_PROGRESS

一个元组，若其t_xmin状态为IN_PROGRESS，基本上是不可见的（规则3和4），除非满足一种特殊情况。

如果此元组是由其他事务插入的，并且t_xmin的状态为IN_PROGRESS，那么这个元组显然是不可见的（规则4）。

如果t_xmin等于当前事务的txid（即，此元组是由当前事务插入的）并且t_xmax不是INVALID，那么这个元组
是不可见的，因为它已经被当前事务更新或删除了（规则3）。

例外情况是当此元组是由当前事务插入并且t_xmax为INVALID时。在这种情况下，从当前事务的角度看，
此元组必须是可见的（规则2），因为这个元组是当前事务自身插入的元组。

5.6.3. Status of t_xmin is COMMITTED

当元组的t_xmin状态为COMMITTED时，该元组通常是可见的（规则6、8和9），除非满足以下三个条件。

规则6很明显，因为t_xmax是INVALID或ABORTED。接下来描述三个例外条件以及规则8和9：

第一个例外条件是t_xmin在获取的事务快照中是活动的（规则5）。在这种情况下，该元组是不可见的，
因为应将t_xmin视为仍在进行中。

第二个例外条件是t_xmax等于当前txid（规则7）。在这种情况下，如同规则3，该元组是不可见的，
因为它已经被当前事务自身更新或删除。

相比之下，如果t_xmax的状态是IN_PROGRESS且t_xmax不是当前txid（规则8），那么该元组是可见的，
因为它尚未被删除。

第三个例外条件是t_xmax的状态为COMMITTED且在获取的事务快照中t_xmax不活跃（规则10）。
在这种情况下，该元组是不可见的，因为它已被另一个事务更新或删除。

相反，如果t_xmax的状态为COMMITTED，但在获取的事务快照中t_xmax是活动的（规则9），那么该元组是可见的，
因为应将t_xmax视为仍在进行中。

5.7 可见性检查

本节说明了PostgreSQL如何执行可见性检查，这是在给定事务中选取适当版本的堆元组的过程。此外，本节还介绍了PostgreSQL
如何防止ANSI SQL-92标准中定义的几种异常情况：脏读（Dirty Reads）、
可重复读（Repeatable Reads）和幻读（Phantom Reads）。

5.7.1. Visibility Check

在图5.10所示的场景中，SQL命令按照以下时间序列执行：

T1: 开始事务（txid 200）
T2: 开始事务（txid 201）
T3: 执行txid 200和201的SELECT命令
T4: 执行txid 200的UPDATE命令
T5: 再次执行txid 200和201的SELECT命令
T6: 提交txid 200
T7: 执行txid 201的SELECT命令

为简化描述，假设仅有两个事务，即txid 200和txid 201。txid 200的隔离级别为READ COMMITTED，
而txid 201的隔离级别为READ COMMITTED或REPEATABLE READ。

接下来，我们探究SELECT命令是如何对每个元组执行可见性检查的。

我们来探讨T3时刻的SELECT命令是如何对每个元组进行可见性检查的。

在T3时刻，表’tbl’中仅有Tuple_1这一条元组，并且根据规则6，Tuple_1对所有事务都是可见的。
因此，txid 200和txid 201的SELECT命令都会返回’Jekyll’。

T5时刻的SELECT命令执行情况如下：

首先，考察txid 200执行的SELECT命令。根据规则7，Tuple_1对txid 200是不可见的，
因为Tuple_1已被同一事务内的UPDATE操作所修改；而Tuple_2根据规则2是可见的，因为它是txid 200自己的
插入操作产生的。因此，txid 200的这次SELECT命令会返回’Hyde’。

另一方面，在txid 201执行的SELECT命令中，Tuple_1根据规则8是可见的，
因为它虽然被其他事务（txid 200）更新，但更新事务并未提交；而Tuple_2根据规则4是不可见的，
因为txid 201在其事务快照中认为txid 200的修改尚未发生。因此，txid 201的这次SELECT命令
会返回’Jekyll’。

如果在事务提交之前，其他事务能够看到被更新的元组，这种情况被称为脏读（Dirty Reads），
也称为写-读冲突（wr-conflicts）。然而，如上所示，在PostgreSQL的任何隔离级别中都不会发生脏读。

T7时刻的SELECT命令：

接下来，描述T7时刻的SELECT命令在两种隔离级别下的行为。

当txid 201处于READ COMMITTED隔离级别时，由于事务快照为‘201:201:’，所以txid 200被视为已提交。
因此，根据规则10，Tuple_1对txid 201不可见；而根据规则6，Tuple_2对txid 201是可见的。这次SELECT命令的
结果是返回’Hyde’。

需要注意的是，在txid 200提交前后执行的SELECT命令得到的结果不同。
这通常被称为不可重复读（Non-Repeatable Reads）现象。

相反，当txid 201处于REPEATABLE READ隔离级别时，因为其事务快照为‘200:200:’，
所以必须将txid 200视为仍在进行中。因此，根据规则9，Tuple_1对txid 201是可见的；
而根据规则5，Tuple_2对txid 201是不可见的。这次SELECT命令的结果是返回’Jekyll’。

请注意，在REPEATABLE READ（以及SERIALIZABLE）隔离级别下，不会发生不可重复读现象。

Hint Bits

为了获取事务的状态，PostgreSQL内部提供了三个函数：TransactionIdIsInProgress、
TransactionIdDidCommit和TransactionIdDidAbort。这些函数设计来减少对事务状态日志（clog）频繁的访问，
可能会利用缓存等机制来优化。但是，如果每次检查元组时都要执行这些函数，仍然可能会成为瓶颈。

为了解决这个问题，PostgreSQL采用了提示位（hint bits）机制，具体如下：

在读取或写入元组时，如果可能，PostgreSQL会在元组的t_infomask中设置提示位。例如，
假设PostgreSQL检查了一个元组的t_xmin状态，并发现其为COMMITTED。此时，PostgreSQL就会在该元组的
t_infomask中设置HEAP_XMIN_COMMITTED提示位。一旦设置了提示位，之后对于该元组的t_xmin来说，
就不再需要调用TransactionIdDidCommit和TransactionIdDidAbort函数了。因此，PostgreSQL能够高效地
检查每个元组的t_xmin和t_xmax状态。

提示位机制显著提高了处理事务状态验证的效率，减少了对clog的直接访问需求，从而提升了数据库的整体性能。

5.8 防止丢失更新

丢失更新（又称写-写冲突）是指并发事务同时更新同一行数据时出现的一种异常情况，
必须在可重复读（REPEATABLE READ）和可串行化（SERIALIZABLE）隔离级别中防止此类问题的发生。
（需要注意的是，读已提交（READ COMMITTED）隔离级别并不需要防止丢失更新。）
本节将阐述PostgreSQL如何防止丢失更新现象，并给出相应示例。

5.8.1. Behavior of Concurrent UPDATE Commands

当执行UPDATE命令时，内部会调用函数ExecUpdate。以下是ExecUpdate的伪代码表示：

Pseudocode: ExecUpdate:

(1) 获取此UPDATE命令将要更新的每一行。

(2) 重复以下过程，直到目标行被更新（或当前事务被中止）。

(3) 如果目标行正在被其他事务更新，则转到步骤(3)；否则，转到步骤(8)。

(4) 等待更新目标行的事务终止，因为在SI（Serializable Isolation）中，
PostgreSQL采用先更新者获胜的策略。

(5) 如果更新目标行的事务状态为COMMITTED，并且当前事务的隔离级别为
REPEATABLE READ（或SERIALIZABLE），则转到步骤(6)；否则，转到步骤(7)。

(6) 中止当前事务以防止丢失更新。

(7) 转到步骤(2)，并在下一轮尝试再次更新目标行。

(8) 如果目标行已被其他并发事务更新，则转到步骤(9)；否则，转到步骤(12)。

(9) 如果当前事务的隔离级别为READ COMMITTED，则转到步骤(10)；否则，转到步骤(11)。

(10) 更新目标行，然后转到步骤(1)。

(11) 中止当前事务以防止丢失更新。

(12) 更新目标行，然后转到步骤(1)，因为目标行尚未被修改或已被终止的事务更新，
即存在写-写冲突（ww-conflict）。

此函数依次为每个待更新的目标行执行更新操作。它有一个外层循环来更新每一行，
而内部while循环则包含了三个分支，分支条件如图5.11所示。

[1] 目标行正在被更新（图5.11[1]）
“正在被更新”意味着该行正被另一个并发事务更新，且该事务还未终止。
在这种情况下，当前事务必须等待更新目标行的事务终止，因为PostgreSQL的SI（Serializable Isolation）
采用先更新者获胜的策略。
例如，假设事务Tx_A和Tx_B并发运行，Tx_B尝试更新一行，但Tx_A已经更新了它且还在进行中。
在这种情况下，Tx_B会等待Tx_A终止。在更新目标行的事务提交后，当前事务的更新操作将继续进行。
如果当前事务处于READ COMMITTED隔离级别，目标行将被更新；
否则（在REPEATABLE READ或SERIALIZABLE级别），当前事务会立即中止以防止丢失更新。

[2] 目标行已被并发事务更新（图5.11[2]）
当前事务尝试更新目标元组，但另一个并发事务已经更新了目标行并已提交。
在这种情况下，如果当前事务处于READ COMMITTED隔离级别，目标行将被更新；
否则，当前事务会立即中止以防止丢失更新。

[3] 无冲突（图5.11[3]）
当没有冲突时，当前事务可以直接更新目标行。

First-updater-win / First-commiter-win
如本节所述，PostgreSQL基于SI（Serializable Isolation）的并发控制采用了“先更新者获胜”的策略来避免丢失更新异常。
相比之下，正如下一节所解释的，
PostgreSQL的SSI（Serializable Snapshot Isolation）采用的是“先提交者获胜”的策略来避免序列化异常。

5.8.2. Examples

下面将通过三个例子进行说明。前两个例子展示了目标行正在被更新时的行为，
第三个例子展示了目标行已被更新时的行为。

5.8.2.1. Example 1

Transactions Tx_A and Tx_B update the same row in the same table,
and their isolation level is READ COMMITTED.

Tx_B的执行过程如下：

执行UPDATE命令后，Tx_B应当等待Tx_A终止，因为目标元组正被Tx_A更新（遵循ExecUpdate中的步骤(4)）。
Tx_A提交后，Tx_B尝试更新目标行（遵循ExecUpdate中的步骤(7)）。
在ExecUpdate的下一轮循环中，目标行被Tx_B再次更新（遵循ExecUpdate中的步骤(2),(8),(9),(10)）。

5.8.2.2. Example 2

Tx_A and Tx_B update the same row in the same table, and their isolation levels
are READ COMMITTED and REPEATABLE READ, respectively.

Tx_B的行为描述如下：

执行UPDATE命令后，Tx_B需要等待Tx_A终止（遵循ExecUpdate中的步骤(4)）。
在Tx_A提交后，由于目标行已被更新，并且当前事务Tx_B的隔离级别为REPEATABLE READ，
因此Tx_B会被中止以解决冲突（遵循ExecUpdate中的步骤(5)和(6)）。

5.8.2.3. Example 3

Tx_B（处于REPEATABLE READ隔离级别）尝试更新已被提交的Tx_A更新过的目标行。
在这种情况下，Tx_B将会被中止（遵循ExecUpdate中的步骤(2),(8),(9)和(11)）。

5.10 必要的维护过程

PostgreSQL的并发控制机制需要以下维护进程:

删除死元组和指向相应死元组的索引元组。
删除日志文件(clog)中不必要的部分。
冻结旧的事务ID(txid)。
更新文件段映射(FSM)、可见性映射(VM)和统计信息。

前两个过程的需求已分别在第5.3.2节和第5.4.3节中进行了解释。
第三个过程与事务ID环绕问题有关,将在以下小节中简要描述。

在PostgreSQL中,VACUUM处理负责这些过程,并在第6章中进行了描述。

5.10.1. FREEZE Processing

这里我描述了事务ID(txid)环绕问题。

假设元组Tuple_1以txid 100插入,即Tuple_1的t_xmin为100。

服务器已运行很长时间,Tuple_1未被修改。当前txid为21亿+100,执行一条SELECT命令。
此时,Tuple_1可见,因为txid 100在过去。然后再次执行相同的SELECT命令,此时当前txid为21亿+101。
但是,Tuple_1不再可见,因为txid 100在未来(图5.20)。

这就是PostgreSQL中所谓的事务ID环绕问题。

为了解决这个问题,PostgreSQL引入了一个称为
“冻结事务ID(frozen txid)”的概念,并实现了一个称为”FREEZE”的进程。

在PostgreSQL中,冻结事务ID是一个特殊的保留事务ID 2,它始终比所有其他事务ID更旧。换句话说,
冻结事务ID始终处于非活动状态并可见。

FREEZE进程由VACUUM进程调用。FREEZE进程扫描所有表文件,如果t_xmin值比当前事务ID减去
vacuum_freeze_min_age(默认为5000万)更旧,就将元组的t_xmin值重写为冻结事务ID(2)。
这在第6章中有更详细的解释。

例如,如图5.21a所示,当前事务ID为5000万,并由VACUUM命令调用FREEZE进程。
在这种情况下,Tuple_1和Tuple_2的t_xmin都被重写为2。

在9.4或更高版本中,XMIN_FROZEN位被设置在元组的t_infomask字段中,
而不是将t_xmin重写为冻结事务ID(图5.21b)。
