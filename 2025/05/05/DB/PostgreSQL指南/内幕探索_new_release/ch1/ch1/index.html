1.1 数据库集簇的逻辑结构

数据库集群是由PostgreSQL服务器管理的一系列数据库的集合。如果您是首次听到这个定义，
可能对此概念感到好奇。在PostgreSQL中，“数据库集群”这个词并不意味着“一组数据库服务器”。
一个PostgreSQL服务器运行在一个单一主机上并管理着一个单独的数据库集群。

图1.1展示了数据库集群的逻辑结构。一个数据库是数据库对象的集合。在关系型数据库理论中，
数据库对象是一种用于存储或引用数据的数据结构。堆表是一个典型的例子，还有许多其他类型，
比如索引、序列、视图、函数等。在PostgreSQL中，数据库自身也是数据库对象，并且在逻辑上相互独立。
所有其他的数据库对象（如表、索引等）都归属于各自的数据库。

PostgreSQL中的所有数据库对象都通过各自的对象标识符（OID）进行内部管理，这些标识符是无符号
的4字节整数。数据库对象及其相应OID之间的关联关系会根据对象的类型存储在相应的系统目录中。
例如，数据库的OID存储在pg_database表中，而堆表（普通表）的OID则存储在pg_class表中。

1.2 数据库集簇的物理结构

数据库集群基本上就是一个单一的目录，称为基本目录（base directory）。它包含一些子目录和
众多文件。当你执行initdb工具初始化一个新的数据库集群时，会在指定的目录下创建一个基本目录。
基本目录的路径通常被设置为环境变量 PGDATA 的值。

图1.2展示了PostgreSQL中数据库集群的一个示例。一个数据库是基本子目录下的一个子目录，
而每个表和索引至少是存储在其所属数据库子目录下的一个文件。还有几个包含特定数据和配置文件的子目录。

虽然PostgreSQL支持表空间（tablespaces），但这一术语的含义与其他关系型数据库管理系统（RDBMS）有所不同。在PostgreSQL中，表空间是一个包含基本目录之外某些数据的单一目录。

在接下来的各个小节中，将详细描述PostgreSQL中数据库集群的结构布局、数据库、与表和索引相关的文件，以及表空间的安排。

1.2.2 数据库的布局

数据库是基础子目录下的一个子目录。数据库目录的名称与其相应的OID（对象标识符）相同。
例如，如果数据库’sampledb’的OID是16384，那么其子目录的名称就是16384。

1.2.4 表空间

PostgreSQL中的表空间是一个位于基本目录之外的额外数据区域。此功能在8.0版本中实现。

图1.3展示了表空间的内部结构及其与主数据区域的关系:

表空间在执行CREATE TABLESPACE语句时指定的目录下创建。在该目录下，会创建一个版本特定的子目录
（例如，PG_14_202011044）。版本特定子目录的命名规则如下所示。

例如，如果在目录‘/home/postgres/tblspc’下创建了一个名为’new_tblspc’的表空间，其OID为16386，则在该表空间下会创建一个名为‘PG_14_202011044’的子目录。

表空间目录通过从pg_tblspc子目录下的符号链接进行寻址。该链接名称与表空间的OID值相同。

如果在该表空间下创建一个新数据库（OID 16387），则其目录将在版本特定的子目录下创建。

如果在基础目录下创建的数据库中创建一个新表，首先会在版本特定的子目录下创建一个新的目录。
新目录的名称与现有数据库的OID相同。接着，新表文件将被放置在所创建的目录下。

1.3 堆表文件的内部布局

在数据文件（堆表、索引、自由空间映射和可见性映射）内部，它被划分为固定长度的页（或块），
默认长度为8192字节（8 KB）。每个文件内的页从0开始顺序编号，这些编号称为块号。如果文件已满，
PostgreSQL会在文件末尾添加一个新的空白页以增大文件尺寸。

页的内部布局取决于数据文件类型。本节将描述表的布局，因为这些信息在后续章节中将会用到。

表内的一页包含三种数据：

  1. 堆元组（heap tuple） - 堆元组是记录数据本身。
    堆元组按照从页底部开始的顺序堆叠。 堆元组的内部结构在第5.2节和第9章中有描述，
    因为这需要了解PostgreSQL中的并发控制（CC）和预写日志（WAL）知识。

  2. 行指针（line pointer） - 行指针长4字节，持有指向每个堆元组的指针。也被称为项目指针。
    行指针形成一个简单的数组，起到对元组的索引作用。每个索引从1开始编号，称为偏移量编号。
    当新元组添加到页上时，也会在数组中新增一个行指针指向新元组。

  3. 页头数据（header data） - 由PageHeaderData结构定义的页头数据位于页的开头，占24字节，
    包含了关于该页的一般信息。 结构的主要变量描述如下：

PageHeaderData

pd_lsn - 此变量存储最后一次更改此页时所写的XLOG记录的LSN（日志序列号）。
它是一个8字节的无符号整数，与WAL（预写日志）机制有关。详情在第9.1.2节描述。

pd_checksum - 此变量存储该页的校验和值。（请注意，此变量在9.3版本或之后支持；在
早期版本中，这部分存储的是页的时间线ID。）

pd_lower, pd_upper - pd_lower指向行指针的结束位置，pd_upper指向最新堆元组的起始位置。

pd_special - 此变量用于索引。在表的页中，它指向页的末尾。（在索引的页中，它指向特殊空间的起始，
特殊空间是仅由索引持有的数据区域，根据索引类型（如B树、GiST、GiN等）包含特定数据。）

页内行指针结束与最新元组起始之间的空隙被称为自由空间或空洞。

为了在表内唯一标识一个元组，内部使用了元组标识符（TID）。TID由一对值组成：包含该元组的页的块号，
以及指向该元组的行指针的偏移量编号。其典型应用场景之一是索引。更多细节请见第1.4.2节。

提示

The structure PageHeaderData is defined in src/include/storage/bufpage.h.

在计算机科学领域，这种类型的页被称为槽页（slotted page），行指针则对应于一个槽数组（slot array）。

此外，大小超过约2KB（8KB的大约1/4）的堆元组将使用一种称为TOAST（The Oversized-Attribute Storage Technique，大属性存储技术）
的方法进行存储和管理。详细信息请参考PostgreSQL文档。

1.4.2 读取堆元组的过程

这里概述两种典型的访问方法：顺序扫描和B-树索引扫描：

顺序扫描 – 通过遍历每个页中的所有行指针，顺序读取所有页中的所有元组。参见图1.6(a)。

B-树索引扫描 – 读取包含索引元组的索引文件，每个索引元组由索引键和指向目标堆元组的TID组成。
如果找到了具有所需键的索引元组，PostgreSQL将使用获得的TID值读取目标堆元组。(这里未解释如何在B-树索引中查找索引元组的具体方法，
因其过程较为常见且篇幅有限。详情请参阅相关资料。) 例如，在图1.6(b)中，
获取的索引元组的TID值为‘(块 = 7, 偏移 = 2)’。这意味着目标堆元组位于表中第7页的第2个元组，
因此PostgreSQL可以直接读取所需的堆元组，无需在其他页中进行不必要的扫描。
